The simulation framework JAMES II.

General description

JAMES II is a general and open modeling and simulation framework based on the "Plug'n simulate" concept.
In principal every modeling formalism / every language can be integrated into the framework. The same holds true for every possible execution mechanism / algorithm.

The framework is distributed under the JAMESLIC (see below)

Thus the framework can be used for

  • discrete
    • stepwise
    • event
  • continuous
  • hybrid
modelling and simulation.
Thereby neither the number of modelling paradigms supported nor the number of simulation algorithm per paradigm is limited.
The framework does not provide integrated support for multi - paradigm modelling (aka multi - formalism modelling), i.e., the combination of a set of model parts (components, building blocks) based on different paradigms to a new model. However, from the point of view of the framework, such a multi modelling paradigm does not differ from a pure one: it can be added as new "modelling plug-in", shipping with specialized synchonization simulation algorithms. Whether these depend on further "pure" plug-ins or not is up to the developer.

The framework has been designed to support

  • sequential
  • parallel
    • coarse-grained
    • fine-grained
execution of simulations.

The benefits of using the framework are manifold, among these:

  • Reuse of central infrastructure, e.g., experiment definition and execution, data collection, model optimization, coarse grained execution, of data structures, etc ... and thus save a lot of development and testing time
  • Reducing the amount of overall work to be done
    • select one of the available modelling paradigms and immediately start to model (without the need to create your own tool first)
    • add a new modelling paradigm, but rely on the experiment stuff, the distributed management, and so on
  • Make your results available and comparable, especially any benchmark results. Compare your algorithms to algorithms provided by others.
  • Reuse of functionality, thus turning the framework into a "library".

Check www.jamesii.org for news around the framework.

About the core

The core of the framework comprises the plug-in management, the service management, experiment control, basic instrumentation and data collection, random number generation, as well as many commonly used extension points most often together with some plug-ins for these (to have a out of the box system).
In addition a flexible GUI is shipped with the core. This GUI can be, but has not to be used to create your own view on the framework. Reusing the GUI has the advantage that a lot of basic logic / dialogs / a.s.o. can be reused. However, if you already have your own GUI you can still integrate the framework in a seamless manner.

How to start

Maybe it is always a good start to execute the framework with the default GUI, and to play around with the menus - just to get an impression what's that all about, what's in there, and what's required from you according to your wishes.

As a modeller

A plug-in for the formalism / language you wanna use exists

Start modelling and then start using the default GUI for checking your model. Finally, if required, you can either adapt the GUI or create your own specialized one before you release the model.

A plug-in does not exist

You have to create interfaces and classes for your executable and symbolic model representation, as well as at least one simulation algorithm. However, you can resue anything provided by the framework on doing so, and you "only" have to program against existing interfaces of the framework. Once you'are done you can start modelling.

As a m&s tool developer

You can use the framework as library, you can make the framework the base /core of your tool, but hide everything, or you can create your application by extending the framework. The latter should be the way with the least effort for your side.

Remarks

The framework is being under development at a research group in a university. Consequently the main purpose of the framework has been knowledge gathering. Already up to now quite a lot of different people with differing programming experience have contributed to the system. Thus the quality of parts of the code might differ. That's one of the reasons for the growing tests package - we at least try to make sure that some basic functionality should be fine (should be - because you never can show that there are no bugs by using tests). In addition these tests might increase the confidence in the code. Nevertheless some parts of the framework may not be as you like them to be. Other parts of the framework might not be as a descriptions depicts, or any publication says: sadly, even though seldomly and always done with great care, parts of the framework have been intensively refactored, and sometimes functionality got lost on doing so. Currently we think that there will not be so much refactoring any more, and thus it should be save to start using the code. These are some of the reasons the framework is labeled as "alpha" version. It is still being under development and things might be changed in the future.

Terms

Modeling and simulation is a domain in which many terms are used quite differently. In this software we decided to stick with older definitions of terms:
  • Simulation
    A simulation is an experiment on a model. Thus during the execution of a simulation the model is computed using different parameters. There might the need of replications per parameter combination.
  • Model
    A model of a system describes the system with a certain purpose.
  • Replication
    A replication is a computation of a model using the same model parameters as for all other replications of the same parameter configuration besides the initialization of random number generators (in the model or of the computation algorithm.)
  • Parameter configuration
    A parameter configuration is a set of parameters for a model instance - it might be a single configuration you are interested in or a configuration computed by an optimization algorithm and so on ...
  • Computation algorithm
    A computation algorithm can be anything from an algorithm to compute a trajectory of a model to model checking.
  • Executable model
    An executable model is an execution efficient representation of a model. Usually it is created from a description, e.g., a modeling language.
  • Symbolic model
    A symbolic model is an (incomplete) model. This representation is used for editing and for model conversions / transformations / compilations.
  • Experiment
    The term "experiment" is used to describe the controlled computation of a number of parameter combinations on a model. Thereby this is not restricted to "simulating" - such an experiment can be used for model checking computations as well.
